## Skill 设计方法论

### 核心原则（借鉴 Specification-Driven Development）

**规格先行理念**（2026-05-20 引入）：
- Skill 不是"代码生成器"，而是"规格执行引擎"
- 维护 Skill = 演进规格文档，而非修补代码
- 每个 Skill 必须有明确的"宪法"（不可变原则）

**模板驱动质量**（约束 LLM 输出）：
1. **防止过早实现细节** — Skill 描述写 WHAT/WHY，不写 HOW
2. **强制不确定性标记** — 模糊点必须用 `[NEEDS CLARIFICATION]` 标注
3. **结构化检查清单** — 每个 Skill 必须有自检清单（相当于规格的"单元测试"）
4. **宪法合规门控** — 核心规则必须是决策树，覆盖所有分支
5. **层级化细节管理** — 主文档保持高层可读，细节提取到独立文件
6. **测试优先顺序** — 先定义验证标准，再生成内容
7. **防止投机性功能** — 每个功能必须追溯到具体用户需求

**Skill 宪法（不可变原则）**：
- **Library-First** — 每个 Skill 必须是独立可复用的模块
- **CLI Interface** — 所有功能必须可通过文本输入/输出验证
- **Test-First** — 先定义质量标准和验证方法，再执行
- **Simplicity** — 初始实现最多 3 个主要步骤，禁止过度设计
- **Anti-Abstraction** — 直接解决问题，不要再包一层抽象
- **Integration-First** — 用真实数据/环境测试，不用 mock

**质量闭环要求**：
- **新建/升级 Skill 时，必须对照 memory/skill-quality-loop.md 检查质量闭环覆盖度**
- 最低要求：有独立审查环节(A) + 问题severity分级(F) + 致命问题有修复循环(C)
- 报告/论文类 Skill 必须实现 A+C+F，建议实现 B+D+E
- 详见 memory/skill-quality-loop.md 的"Skill设计检查清单"和"按Skill类型的最低配置"

**三层验证标准**（基于 SDD 方法论）：
- **L1 工具调用完整性** — 所有必需步骤是否执行（如图片提取、API调用、验证步骤）
- **L2 实体准确性** — 数据、引用、数值是否正确（如参考文献真实性、品位数据准确性）
- **L3 状态收敛** — 最终交付物是否干净完整（如文件大小、无孤立引用、格式合规）

## Skill 执行完整性自检规则（2026-05-19 固化）

> **核心原则**：大型 skill 执行完成后，必须自检流程完整性，防止跳过关键步骤导致质量问题。

### 适用范围（"大型 skill" 定义）
满足以下任一条件的 skill 必须执行自检：
- **多阶段工作流**：包含 3 个以上明确的执行阶段（如 chinese-paper 的"提取图片→生成内容→插入图片→验证"）
- **多 Agent 协作**：使用 cowork 模式或委托 sub-agent 执行
- **关键质量要素**：涉及数据提取、格式转换、文件生成等容易遗漏的步骤
- **历史跳过记录**：该 skill 曾因跳过步骤导致质量问题

**典型适用 skill**：chinese-paper, mining-investment-report, copper-ore, gold-ore, mineral-system-eval, technical-monograph 等

### 自检清单机制

**执行完成后立即自检（3 分钟内）：**

1. **流程完整性检查**：
   - 对照 skill 定义的标准流程，逐项确认是否执行
   - 检查是否有"应该做但实际跳过"的步骤
   - 特别关注：数据提取、格式转换、质量验证等容易遗漏的环节

2. **关键产物验证**：
   - 检查必需的中间产物是否生成（如图片文件、数据表格、临时文件）
   - 检查最终交付物的完整性指标（如文件大小、图片数量、章节数量）
   - 对照 skill 定义的质量标准，确认是否达标

3. **跳过原因分析**：
   - 如果发现跳过步骤，分析原因：技术限制？逻辑错误？遗忘？
   - 判断跳过是否影响质量（轻微/中等/严重）

### 补救流程

**发现跳过步骤时的处理：**

```
IF 跳过步骤 AND 影响质量:
  1. 立即告知用户："检测到跳过了 [步骤名称]，正在补救"
  2. 执行被跳过的步骤
  3. 重新验证最终产物
  4. 更新交付物
  5. 向用户确认："已补救完成，请验证"

IF 跳过步骤 BUT 不影响质量:
  1. 记录到执行日志（不打断用户）
  2. 在 skill-metrics.md 中标注"部分流程跳过但不影响质量"
```

### 预防跳过机制

**设计层面（写 skill 时）：**
1. **强制检查点**：在 skill 中明确标注"不可跳过"的步骤，用 ⚠️ 或 🔴 标记
2. **前置条件验证**：每个阶段开始前检查上一阶段的产物是否存在
3. **流程清单**：在 skill 开头列出完整流程清单，执行时逐项打勾

**执行层面（跑 skill 时）：**
1. **委托 Agent 时明确要求**：在 prompt 中列出完整流程清单，要求"必须全部执行"
2. **中间产物强制保存**：每个阶段的产物保存到临时文件，便于验证
3. **进度追踪**：使用 TaskList 或 checkpoint 机制追踪执行进度
